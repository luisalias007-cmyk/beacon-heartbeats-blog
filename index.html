Getting Started with Beacon 2.x: Heartbeats for AI Agents

Published June 1, 2026. Tutorial for Beacon 2.6+ concepts, verified with a local signed-heartbeat example.

Beacon is an agent coordination protocol: a compact way for autonomous or semi-autonomous agents to say who they are, what they are doing, whether they are alive, and what kind of collaboration they want. The official beacon-skill repository describes Beacon as the social and economic coordination layer that sits beside task protocols and tool protocols. In practice, that means Beacon is useful for the part of an agent system that usually gets improvised: presence, liveness, discovery, trust, status, and small coordination messages.

This tutorial focuses on a heartbeat. A heartbeat is intentionally boring, and that is why it is valuable. If an agent can publish a signed heartbeat every few seconds or minutes, other agents can answer simple operational questions: is the indexing worker still alive, did the research agent go idle, did the scheduler change state, and when should a human or fallback agent intervene? Beacon 2.6 and later make this pattern concrete by wrapping messages in signed envelopes with an agent identity, a nonce, and a timestamp. The signature gives peers a way to distinguish a real update from a copied or forged status line.

The Mental Model

Think of a Beacon message as three layers. The payload says what happened: for example, kind: heartbeat, status: alive, or kind: mayday. The identity says which agent is speaking. The signature proves that the holder of the private key approved the payload. Transport is separate. You can move the envelope through a webhook, a local UDP bus, a social-style feed, a task board, or a RustChain-related channel. That separation keeps the message portable.

This is different from a task delegation protocol. A task protocol might say "summarize this repository" or "run this tool". Beacon says "this agent exists, here is its current status, here is how to reach it, and here is a signed coordination signal." In a multi-agent system, both are needed. Tools make agents capable; heartbeats make them observable.

Install The Official Package

On Linux, macOS, or WSL, install the official CLI from PyPI: 

python3 -m venv .venv
. .venv/bin/activate
pip install beacon-skill
beacon identity new
beacon webhook serve --port 8402

In another terminal, send a local loopback message: 

. .venv/bin/activate
beacon webhook send http://127.0.0.1:8402/beacon/inbox --kind hello --text "Hello from my agent"
beacon inbox list --limit 1

Verification note: on my Windows test environment, beacon-skill==2.16.0 installed successfully, but the CLI imported Python's Unix-only fcntl module and failed before startup. The commands above follow the official README and should be run on Linux/macOS/WSL until that Windows import path is fixed.

A Copy-Paste Heartbeat Example

The following Python script is self-contained and runs on Windows, Linux, or macOS after installing cryptography. It does not replace the official Beacon CLI. Instead, it demonstrates the core Beacon heartbeat idea: derive an agent id from an Ed25519 public key, create a canonical JSON payload, sign it, and verify the envelope before trusting it. 

python -m pip install cryptography
import base64
import hashlib
import json
import secrets
import time

from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey
from cryptography.hazmat.primitives.serialization import Encoding, PublicFormat


private_key = Ed25519PrivateKey.generate()
public_key = private_key.public_key()
public_bytes = public_key.public_bytes(Encoding.Raw, PublicFormat.Raw)

agent_id = "bcn_" + hashlib.sha256(public_bytes).hexdigest()[:12]

payload = {
    "kind": "heartbeat",
    "agent_id": agent_id,
    "status": "alive",
    "service": "retrieval-worker",
    "timestamp": int(time.time()),
    "nonce": secrets.token_hex(8),
}

canonical = json.dumps(payload, sort_keys=True, separators=(",", ":")).encode()
signature = private_key.sign(canonical)

envelope = {
    "beacon_version": "2.x",
    "payload": payload,
    "public_key": base64.b64encode(public_bytes).decode(),
    "signature": base64.b64encode(signature).decode(),
}

print(json.dumps(envelope, indent=2))

# Receiver side: verify before acting on the heartbeat.
public_key.verify(signature, canonical)
print("verified heartbeat from", agent_id)

When this script prints verified heartbeat from bcn_..., you have the smallest useful heartbeat loop. A production receiver would add replay protection by storing recent nonces, reject stale timestamps, and map the status to an operational decision. For example, if a heartbeat has not arrived in five minutes, a coordinator could send a Beacon mayday message, pause new assignments, or ask a backup worker to take over.

From Heartbeats To Contracts

A heartbeat tells the network that an agent is alive. A contract tells the network what the agent is responsible for. For a small system, that contract can be a JSON document: 

{
  "contract_id": "daily-index-2026-06-01",
  "agent_id": "bcn_example1234",
  "role": "retrieval-worker",
  "inputs": ["repo_url", "branch"],
  "outputs": ["index_manifest", "status_report"],
  "heartbeat_interval_seconds": 60,
  "mayday_after_seconds": 300
}

The important part is not the exact schema. The important part is that the contract and the heartbeat agree. If the contract says a worker should check in every 60 seconds, a coordinator can treat silence as a measurable failure instead of a vague feeling that "the agent got stuck." Beacon turns the invisible state of agent coordination into signed, inspectable messages.

Mayday Signals

Heartbeats are normal operation. Mayday signals are the exception path. A good mayday message should be short, signed, and actionable. It should say which agent is blocked, what changed, what help is needed, and whether the task can be safely retried. A useful pattern is: 

{
  "kind": "mayday",
  "agent_id": "bcn_example1234",
  "task": "daily-index-2026-06-01",
  "reason": "upstream rate limit",
  "safe_to_retry": true,
  "next_attempt_after": "2026-06-01T19:30:00Z"
}

That mayday can be routed through the same transport as a heartbeat. The receiving coordinator does not need to understand every internal detail of the blocked agent. It only needs to verify the signature, check freshness, and apply policy: wait, retry, escalate, or reassign.

Where Beacon Fits

Beacon is most useful when agents are not all running inside one process. A single orchestrator can keep state in memory, but a distributed system needs messages that survive process boundaries and transport changes. A signed Beacon envelope can move from a local LAN test to a webhook receiver to a public agent card without changing the core payload.

The design also keeps the human operator in the loop. An operator can inspect a heartbeat, see which agent signed it, read the contract, and understand whether a mayday deserves attention. That matters because the hardest failures in agent systems are often coordination failures: duplicate work, stale assumptions, invisible timeouts, and unclear ownership.

  4. Write one contract for an agent you already use, then define when that agent should send a mayday.

The simplest useful Beacon deployment is not a grand multi-agent city. It is one agent, one signed heartbeat, one receiver that verifies it, and one policy for what happens when the heartbeat stops. Start there, and the rest of the coordination layer has somewhere solid to grow.
